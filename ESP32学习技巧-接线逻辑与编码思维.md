# ESP32 学习技巧：接线逻辑与编码思维

> 基于 ST7789 显示屏调试经验总结的通用方法论，适用于所有 ESP32 外设开发。  
> 涵盖：接线排查法、编码思维模型、总线通信原理、显存工作机制、分层学习路径。

---

## 一、接线逻辑："三层排查法"

遇到硬件不工作时，不要随机尝试，按层级从上到下排查：

| 层级 | 检查项 | 工具 | 思维要点 |
|------|--------|------|---------|
| **物理层** | VCC/GND 是否正常、线是否接反/松动 | 万用表 | "没有 GND = 没有信号回路" |
| **协议层** | SPI/I2C 时序是否正确、CS/DC 状态 | 逻辑分析仪 | "时钟线告诉对方'现在读'，数据线在那一刻才有意义" |
| **应用层** | 地址/坐标/参数值是否正确 | 串口打印 | "应用层错误不会导致'没反应'，只会导致'结果不对'" |

**关键认知**：如果设备"完全没反应"（屏幕不亮、传感器无数据），问题一定在**物理层或协议层**；如果"有反应但结果不对"（花屏、数据偏差），问题在**应用层**。

**为什么从物理层开始？**

因为物理层是所有信号的基础。没有正确的电压和回路，协议层和应用层都不存在。就像打电话：如果电话线断了（物理层），讨论拨号方式（协议层）或说话内容（应用层）都没有意义。

## 二、编码思维："先配置，后使用"

所有外设都遵循同样的初始化模式：

```
使用任何外设的三步：
1. 初始化（配置工作模式：退出睡眠、设置格式）
2. 设置参数（地址范围、坐标原点等）
3. 传输数据（真正干活：写颜色、读传感器）
```

**ST7789 显示屏的对应关系**：
- 初始化：`tft.init(240, 240)` → 发送 0x11/0x36/0x3A/0x29 等命令
- 设置参数：`setRotation(0)` → 发送 0x36 配置行列映射
- 传输数据：`fillRect(...)` → 发送 0x2A/0x2B/0x2C + 颜色数据

**GY-91 传感器的对应关系**：
- 初始化：`Wire.begin(SDA_PIN, SCL_PIN)` → 启动 I2C 总线
- 设置参数：`bmp.begin(0x76)` → 确认设备在线、设置采样精度
- 传输数据：`bmp.readTemperature()` → 读取温度寄存器值

**为什么必须按这个顺序？**

因为外设芯片上电后处于**随机状态**（就像电脑刚开机内存里是垃圾数据）。不先配置，芯片不知道"数据格式是什么"、"行列怎么映射"、"采样精度是多少"，直接发数据它根本无法解析。

**常见错误**：
```cpp
// 错误：没有初始化直接读取
float temp = bmp.readTemperature();  // 可能返回 0 或随机值

// 正确：先初始化再使用
Wire.begin(3, 4);
if (!bmp.begin(0x76)) {
    Serial.println("BMP280 not found!");
    return;
}
float temp = bmp.readTemperature();
```

## 三、总线思维的"点名制"

当你的多个设备共用数据线（如 ST7789 + 字库共用 SDA/SCL），必须理解 **CS 片选的本质是"点名"**：

```
总线结构：

         ESP32
           │
    SDA ───┼────────┬────────┐
           │        │        │
    SCL ───┼────────┼────────┤
           │        │        │
    CS1 ───┴────────┤        │    ← "屏幕，轮到你了"
           │        │        │
    CS2 ────────────┴────────┤    ← "字库，轮到你了"
           │        │        │
        ST7789   GT30L32S4W
        (屏幕)    (字库)
```

**关键规则**：
- 任何时候**只能有一个 CS 为低电平**
- CS 拉低前，确保另一个 CS 已经拉高
- 使用 `SPI.beginTransaction()` / `endTransaction()` 保护临界区

**为什么需要点名制？**

因为 SDA/SCL 是**广播线**——所有挂在总线上的设备都能听到数据。如果没有 CS，两个设备会同时响应，数据就会冲突（就像两个人同时接电话）。

**I2C 的"点名"机制**：

I2C 没有 CS 线，它用**地址**来点名：
```
ESP32: "地址 0x68 在吗？"
MPU9250（地址0x68）: "在！"
BMP280（地址0x76）:  （保持沉默）
```

每个 I2C 设备有唯一的 7 位地址，主设备发送地址后，只有匹配的设备会应答。

## 四、显存思维的"延迟刷新"

你写入的颜色**不是立刻显示**的，而是写入芯片内部的 GRAM（显存），等下一次刷新周期（通常 60Hz = 16.6ms）才会显示。

```
代码执行 fillRect() ──▶ 写入 GRAM ──▶ 16.6ms 后 ──▶ 液晶刷新 ──▶ 肉眼看到
        ↑                                                    ↑
    "立刻完成"                                          "延迟显示"
```

**为什么人眼感觉不到延迟？**

因为 16.6ms 太短（人眼暂留约 100ms）。但如果你在做高速动画，需要知道：GRAM 写入和屏幕刷新是**异步**的。

**关键认知**：
- `fillRect()` 返回时，GRAM 已经更新，但液晶面板还没变
- 液晶面板的刷新由 ST7789 内部定时器控制，和 ESP32 无关
- 如果你连续快速写 GRAM，屏幕只会显示最后一次刷新的内容

## 五、从"接线"到"像素"的完整思维链条

```
1. 【物理层】接好 VCC/GND，芯片上电
2. 【物理层】拉高 BLK，LED 背光亮起
3. 【协议层】tft.init() 发送初始化命令 → 芯片退出睡眠、配置格式
4. 【协议层】fillScreen() 发送颜色数据 → 填满 GRAM
5. 【芯片层】ST7789 每 16.6ms 读取 GRAM → 驱动液晶分子
6. 【物理层】液晶分子改变透光角度
7. 【视觉层】你看到画面
```

**debug 时定位层级**：
- 背光不亮 → 第 2 层（检查 BLK 接线）
- 背光亮但全白 → 第 3 层（检查 CS/DC/SPI 通信）
- 有画面但花屏 → 第 4 层（检查像素格式配置）
- 位置不对 → 第 5 层（检查 setRotation/坐标）

**通用排查流程**：
```
设备不工作？
  ├── 量 VCC（应该是 3.3V 或 5V）
  ├── 量 GND（应该是 0V，且和 ESP32 共地）
  ├── 检查片选/地址（CS 是否拉低？I2C 地址对不对？）
  ├── 用逻辑分析仪抓波形（SPI/I2C 时序是否正常？）
  └── 检查初始化顺序（是不是先 init 再使用？）
```

## 六、推荐学习路径（基于你的硬件）

### 第 1 层：GPIO 基础（1-2 天）

**目标**：理解 ESP32 最基本的输入输出

- `pinMode()` / `digitalWrite()` / `digitalRead()`
- PWM 调光：`ledcSetup()` / `ledcWrite()`
- 延时函数：`delay()` vs `millis()`（非阻塞编程）
- **实战目标**：让板载 RGB LED（GPIO38）变色闪烁

**为什么从 GPIO 开始？**

因为 GPIO 是所有外设的基础。SPI 的 CS/DC 是 GPIO，I2C 的 SDA/SCL 也是 GPIO。理解 GPIO 的输入/输出/上拉/下拉，才能理解更复杂的通信协议。

**关键概念**：
```cpp
pinMode(5, OUTPUT);      // 设置 GPIO5 为输出模式
 digitalWrite(5, HIGH);   // 输出高电平（3.3V）
 digitalWrite(5, LOW);    // 输出低电平（0V）
 
 pinMode(4, INPUT_PULLUP); // 设置 GPIO4 为输入，启用内部上拉电阻
 int state = digitalRead(4); // 读取引脚电平（HIGH 或 LOW）
```

---

### 第 2 层：SPI 通信（2-3 天）

**目标**：理解高速串行通信的原理

- SPI 的四根线：MOSI、MISO、SCK、CS
- 时钟极性和相位（CPOL/CPHA）
- 用逻辑分析仪抓波形（观察 SCK 和 MOSI 的时序关系）
- **实战目标**：让 ST7789 显示自定义图案

**为什么 SPI 比 I2C 快？**

- I2C 是半双工（同一根线既要发又要收），且每次传输都需要发送地址
- SPI 是全双工（MOSI 和 MISO 同时工作），且用 CS 线直接选设备
- SPI 时钟可达 40MHz+，I2C 通常 400kHz

**关键代码**：
```cpp
SPI.begin(12, 13, 11);  // SCK, MISO, MOSI
SPI.beginTransaction(SPISettings(8000000, MSBFIRST, SPI_MODE0));
digitalWrite(CS_PIN, LOW);
SPI.transfer(0x01);     // 发送命令
digitalWrite(CS_PIN, HIGH);
SPI.endTransaction();
```

---

### 第 3 层：I2C 通信（1-2 天）

**目标**：理解地址寻址和寄存器读写

- I2C 的两根线：SDA、SCL
- 设备地址和寄存器地址的区别
- 用 I2C Scanner 扫描总线上的设备
- **实战目标**：读出 GY-91 的加速度/温度数据

**关键概念**：
```cpp
// 写寄存器
Wire.beginTransmission(0x68);  // 开始通信，地址 0x68
Wire.write(0x6B);              // 指定寄存器（电源管理）
Wire.write(0x00);              // 写入值（退出睡眠）
Wire.endTransmission();        // 结束通信

// 读寄存器
Wire.beginTransmission(0x68);
Wire.write(0x3B);              // 指定寄存器（加速度 X 高字节）
Wire.endTransmission(false);   // 不发送停止位
Wire.requestFrom(0x68, 6);     // 读取 6 字节（加速度 XYZ）
while (Wire.available()) {
    byte data = Wire.read();
}
```

---

### 第 4 层：I2S 音频（3-5 天）

**目标**：理解数字音频的传输原理

- I2S 的三根线：BCLK、WS、DATA
- 采样率、位深度、声道数的含义
- DMA 缓冲和双缓冲机制
- **实战目标**：让 MAX98357A 播放 WAV 音频

**为什么 I2S 需要 DMA？**

因为音频数据量很大（44.1kHz × 16bit × 2ch = 1.41Mbps），CPU 逐字节发送会占用 100% 时间。DMA（直接内存访问）让硬件自动从内存搬运数据到 I2S 接口，CPU 只需要准备下一批数据。

**关键概念**：
```cpp
// DMA 缓冲配置
.dma_buf_count = 8,     // 8 个缓冲区
.dma_buf_len = 128,     // 每个缓冲区 128 个采样
// 总缓冲 = 8 × 128 × 4 字节 = 4KB（双声道 16bit）
```

---

### 第 5 层：全双工音频（2-3 天）

**目标**：理解同时录音和播放的原理

- TX（发送）和 RX（接收）共用 BCLK/WS
- 数据方向：GPIO1（输出到功放）、GPIO2（输入从麦克风）
- PSRAM 用于存储录音数据
- **实战目标**：INMP441 录音 → 存 PSRAM → MAX98357A 播放

**为什么共用 BCLK/WS 不会冲突？**

因为 BCLK 和 WS 是**时钟信号**（由 ESP32 产生，所有设备同步），不是数据信号。就像乐队指挥的指挥棒，所有人看着同一个节拍，但各自演奏不同的乐器。

---

### 第 6 层：多模块整合（持续）

**目标**：把多个外设组合成完整项目

- **项目示例**：带 GUI 的音乐播放器
  - ST7789 显示专辑封面和播放进度
  - GY-91 检测姿态（摇一摇切歌）
  - MAX98357A 播放音乐
  - INMP441 语音控制（"播放"、"暂停"）

**整合时的注意事项**：
- GPIO 冲突检查（你的硬件布局已经避免了）
- 中断优先级管理（I2S DMA 中断优先级要高）
- 内存管理（PSRAM 用于大数组，内部 RAM 用于频繁访问的数据）
- 电源管理（多个模块同时工作时的电流需求）

## 八、常见误区与纠正

### 误区 1："代码跑通了，但屏幕不亮，一定是屏幕坏了"

**纠正**：80% 的"不亮"是接线问题，不是硬件损坏。先量 VCC/GND，再检查 CS/DC 接线。

### 误区 2："初始化顺序无所谓，反正最后都配置了"

**纠正**：有些外设有依赖关系。例如 ST7789 必须先发送 0x11（退出睡眠），等待 120ms 后才能发送其他命令。顺序错了，芯片会忽略命令。

### 误区 3："SPI 速度越快越好"

**纠正**：ST7789 最高支持 60MHz，但实际建议 10-40MHz。速度太快可能导致信号完整性问题（尤其是杜邦线连接时）。

### 误区 4："I2C 地址冲突就换一个地址"

**纠正**：不是所有设备都支持地址切换。买硬件前先查 datasheet，确认地址是否与现有设备冲突。

### 误区 5："DMA 缓冲区越大越好"

**纠正**：DMA 缓冲太大会增加延迟（填满缓冲需要时间）。音频播放建议 `dma_buf_len=128~256`，`dma_buf_count=4~8`。

## 九、核心思维总结

### 1. 接线不是随机的

每根线都有明确职责：
- **电源线**（VCC/GND）：生命线
- **数据线**（MOSI/MISO/SDA）：传递信息
- **时钟线**（SCK/BCLK）：同步节拍
- **控制线**（CS/DC/RES）：选择和配置

### 2. 代码顺序不能乱

```
初始化 → 配置参数 → 传输数据
  ↑        ↑         ↑
 必须     必须      干活
```

### 3. 总线就是"公共电话"

- SPI：用 CS "点名"，同一时刻只能一个人说话
- I2C：用地址 "点名"，广播后等待对应的人应答
- I2S：用 WS "分组"，左右声道分时传输

### 4. 显存和显示不同步

你写的是显存，看的是屏幕，两者之间有一个刷新周期（16.6ms）的延迟。

### 5. 从底层往上 debug

```
不工作？
  → 物理层（电压、接线）
    → 协议层（时序、地址）
      → 应用层（参数、逻辑）
```
