# ESP32-S3 + ST7789 + GT30L32S4W 汉字显示踩坑记录与架构演进

> 本文档总结了从项目开始到最终成功的完整踩坑过程，包括汉字不显示、颜色反色、SPI冲突等问题的根因分析，以及从软件SPI到硬件SPI的架构演进。
>
> 硬件：ESP32-S3-DevKitC-1-N16R8 + ZJY154S10Z0TG01（240×240 IPS 带字库版）
> 字库：GT30L32S4W（GB2312，15×16点阵）
> 环境：Arduino IDE

---

## 一、项目目标

在ST7789 240×240 IPS彩屏上显示中文字符，字库来自外置GT30L32S4W字库芯片。

**看似简单，实则涉及多个复杂子系统：**
- ST7789V显示驱动的初始化与时序
- GT30L32S4W字库芯片的SPI读取
- GB2312编码到字库地址的映射
- 软件SPI与硬件SPI的选择与冲突
- IPS面板的像素极性问题

---

整个调试过程经历了以下阶段的问题：

| 阶段 | 问题 | 现象 | 解决方式 |
|------|------|------|----------|

## 三、问题一：汉字不显示——字库地址计算错误

### 3.1 最初错误的地址计算

```cpp
// ❌ 错误版本1：基地址为0
uint32_t getHanziAddr(uint8_t high, uint8_t low) {
  uint8_t qu  = high - 0xA0;
  uint8_t wei = low  - 0xA0;
  return ((uint32_t)(qu - 16) * 94 + (wei - 1)) * 32;  // 基地址=0？
}
```

**问题**：GT30L32S4W的15×16点阵起始地址是**0x2C9D0**，不是0。

### 3.2 修正后的地址计算

```cpp
// ✅ 正确版本
uint32_t gb2312Addr(uint8_t msb, uint8_t lsb) {
  // Zone 1: 符号区（0xA1A1 ~ 0xA9EF）
  if (msb >= 0xA1 && msb <= 0xA9 && lsb >= 0xA1)
    return ((uint32_t)(msb - 0xA1) * 94 + (lsb - 0xA1)) * 32 + 0x2C9D0;
  
  // Zone 2: 汉字区（0xB0A1 ~ 0xF7FE）
  if (msb >= 0xB0 && msb <= 0xF7 && lsb >= 0xA1)
    return ((uint32_t)(msb - 0xB0) * 94 + (lsb - 0xA1) + 846) * 32 + 0x2C9D0;
    //                                           ↑ 关键！+846偏移
  
  return 0;
}
```

**关键知识点**：
- GT30L32S4W字库将Zone 1（符号区，846个字符）和Zone 2（汉字区，6763个字符）连续存储
- Zone 2的汉字前需要跳过846个符号，所以偏移量要**+846**
- 每个字符占32字节（16行 × 2字节/行）

### 3.3 验证方法

```cpp
uint32_t addr = gb2312Addr(0xD6, 0xD0);  // "中"
Serial.printf("地址: 0x%06X\n", addr);   // 应输出 0x04F7F0
```

## 四、问题二：颜色反色——最棘手的问题

### 4.1 现象

- 屏幕初始化后颜色正常
- 一旦执行字库读取操作，屏幕颜色立即反相
- 白色背景变成黑色，红色变成青色

### 4.2 最初的误解

**误解1：RGB/BGR颜色模式问题**
- 以为ST7789的RGB顺序配置错误
- 实际上ST7789的RGB565格式是固定的，没有配置寄存器可以改

**误解2：需要手动调用`invertDisplay(true)`**
- `tft.init()`已经自动发送了`INVON (0x21)`命令
- 手动再调一次是多余的，不是根本原因

### 4.3 根本原因：软件SPI bit-banging干扰硬件SPI

这是整个项目**最核心、最难定位**的问题。

#### 4.3.1 构造方式决定了SPI模式

```cpp
// 方式A：5参数 = 软件SPI（bit-bang）
Adafruit_ST7789 tft(CS, DC, MOSI, SCLK, RST);
// 内部使用 digitalWrite(GPIO12, HIGH/LOW) 模拟时钟

// 方式B：3参数 = 硬件SPI（SPI外设）
Adafruit_ST7789 tft(CS, DC, RST);
// 内部使用 SPI.transfer()，由ESP32-S3的FSPI外设控制
```

#### 4.3.2 致命的错误：在字库读取中调用`SPI.begin()`

```cpp
// ❌ 致命错误！
void readFont(...) {
    SPI.begin(TFT_SCLK, TFT_MISO, TFT_MOSI);  // ← 这一行毁了所有！
    // ...
}
```

**发生了什么**：
1. `Adafruit_ST7789`使用5参数构造 → 内部用**软件SPI**（bit-bang）
2. `SPI.begin()`将GPIO11/12/13切换给**硬件SPI外设**
3. 切换后，`digitalWrite(GPIO12, ...)`不再控制实际引脚
4. Adafruit的软件SPI从此失效，屏幕无法正确接收命令
5. 更隐蔽的是：ST7789对SCK边沿**极度敏感**，即使CS拉高，SCK上的毛刺仍会被当作额外时钟脉冲
6. 这些毛刺导致ST7789内部状态机位偏移，0x21（INVON）命令被意外覆盖

#### 4.3.3 为什么ST7789对SCK如此敏感？

根据TFT_eSPI作者Bodmer的分析（Issue #163）：
- ST7789工作在**SPI Mode 3**（CPOL=1，CPHA=1）
- CS高电平时，SCK**必须保持高电平**
- 如果CS高电平期间SCK出现下降沿，ST7789会将其解释为额外时钟脉冲
- 导致内部命令解析器位偏移，后续所有命令解析错误

```
时序图：
正常：CS ──────┐    ┌─────────────────┐    ┌──────
               └────┘                 └────┘
     SCK ────────────────────────────────（保持高）

异常：CS ──────┐    ┌─────────────────┐    ┌──────
               └────┘                 └────┘
     SCK ──────────┐        ┌───────────────
                    └────────┘
                    ↑ 这个下降沿被ST7789捕获！
```

#### 4.3.4 解决方案

**方案1（推荐）：软件SPI统一**
- 屏幕和字库都使用软件SPI（bit-bang）
- 完全不调用`SPI.begin()`
- 代码：`st7789_hanzi_demo.ino`

**方案2：硬件SPI统一**
- 屏幕和字库都使用硬件SPI（`SPI.beginTransaction()`）
- 通过不同CS分时复用总线
- 代码：`st7789_hanzi_hwspi.ino`

## 五、问题三：屏幕变空白——`SPI.begin()`的破坏力

### 5.1 现象

- 初始化代码能正常显示英文
- 一旦加入字库读取代码，屏幕立即变白或变黑
- 串口显示字库读取成功（有数据返回）

### 5.2 原因

与反色问题同源：

```cpp
Adafruit_ST7789 tft = Adafruit_ST7789(TFT_CS, TFT_DC, TFT_MOSI, TFT_SCLK, TFT_RST);
// 5参数 = 软件SPI

void readFont(...) {
    SPI.begin(...);  // ← 把GPIO11/12切换给硬件SPI
    // ...
}
```

### 5.3 为什么有时能显示英文？

因为英文显示在`SPI.begin()`之前执行，此时软件SPI还正常。
一旦`SPI.begin()`被调用，所有后续的屏幕操作都失效。

### 5.4 正确做法

**软件SPI版**：
```cpp
// 字库读取也用软件SPI，不用SPI库
void fontSend(uint8_t dat) {
  for (int i = 0; i < 8; i++) {
    digitalWrite(TFT_SCLK, LOW);
    digitalWrite(TFT_MOSI, (dat & 0x80) ? HIGH : LOW);
    dat <<= 1;
    digitalWrite(TFT_SCLK, HIGH);
  }
}

uint8_t fontRecv() {
  uint8_t data = 0;
  for (int i = 0; i < 8; i++) {
    digitalWrite(TFT_SCLK, HIGH);
    digitalWrite(TFT_SCLK, LOW);
    data = (data << 1) | digitalRead(TFT_MISO);
  }
  return data;
}
```

**硬件SPI版**：
```cpp
// 3参数构造 = 硬件SPI
Adafruit_ST7789 tft = Adafruit_ST7789(TFT_CS, TFT_DC, TFT_RST);

// 初始化时调用SPI.begin()
SPI.begin();

// 字库读取也走硬件SPI
tft.drawRGBBitmap(...);  // 内部用SPI.transfer()
fontRead(...);           // 也用SPI.beginTransaction()
```

## 六、问题四：显示效率——从逐点绘制到批量绘制

### 6.1 低效方案：`drawPixel`逐点绘制

```cpp
// ❌ 慢！每次drawPixel都要setAddrWindow + SPI传输
void drawHanziSlow(int16_t x, int16_t y, ...) {
  for (uint8_t row = 0; row < 16; row++) {
    for (uint8_t bit = 0; bit < 8; bit++) {
      tft.drawPixel(x + bit, y + row, ...);      // ← 256次调用
    }
  }
}
```

**性能**：
- 每个汉字256次`drawPixel`
- 每次`drawPixel` = `setAddrWindow` + 2字节SPI传输
- 10个汉字约**200ms**

### 6.2 高效方案：`drawRGBBitmap`批量绘制

```cpp
// ✅ 快！一次性推送256个像素
void drawHanziFast(int16_t x, int16_t y, ...) {
  uint16_t pixBuf[16 * 16];
  
  // 填充像素缓冲
  for (int row = 0; row < 16; row++) {
    // ... 解码点阵到pixBuf ...
  }
  
  // 一次性写入
  tft.drawRGBBitmap(x, y, pixBuf, 16, 16);  // ← 1次调用
}
```

**性能**：
- 每个汉字1次`drawRGBBitmap`
- 内部只需1次`setAddrWindow` + 512字节连续SPI传输
- 10个汉字约**30ms**

### 6.3 原理对比

| 方式 | setAddrWindow次数 | SPI传输次数 | 10汉字耗时 |
|------|------------------|-------------|-----------|
| drawPixel | 256次 | 256次 | ~200ms |
| drawRGBBitmap | 1次 | 1次连续 | ~30ms |
| 提升 | **256倍** | **256倍** | **6.7倍** |

## 七、问题五：编码问题——中文乱码

### 7.1 现象

- 代码中直接写中文，编译后显示乱码
- 或者编译错误

### 7.2 原因

Arduino IDE默认使用UTF-8编码保存文件，但GT30L32S4W字库使用**GB2312编码**。

### 7.3 解决方案

**方案1：使用`\xHH`十六进制转义（推荐）**

```cpp
// 不依赖文件编码，字节值始终正确
drawStr(10, 28,
    "\xD6\xD0\xB9\xFA"   // 中国（GB2312: 0xD6D0 0xB9FA）
    "  "
    "\xC4\xE3\xBA\xC3",  // 你好（GB2312: 0xC4E3 0xBAC3）
    0xFFFF, 0x0000);
```

**方案2：文件以GB2312/ANSI编码保存**
- 在Arduino IDE中直接写中文
- 保存时选择"ANSI"或"GB2312"编码
- 注意：这样保存的代码在不同编辑器中可能显示乱码

## 八、架构演进：从混乱到清晰

### 8.1 阶段1：混乱期（问题百出）

```cpp
// ❌ 混乱的代码结构
Adafruit_ST7789 tft(CS, DC, MOSI, SCLK, RST);  // 软件SPI

void readFont() {
    SPI.begin(...);  // 突然切换到硬件SPI？！
    SPI.transfer(...);
}

void setup() {
    tft.init();
    readFont();  // 调用后屏幕失效
    tft.drawPixel(...);  // 逐点绘制，极慢
}
```

**问题**：
- 软件SPI和硬件SPI混用
- 地址计算错误
- 绘制方式低效
- 没有反色处理

### 8.2 阶段2：软件SPI统一期（可用但慢）

```cpp
// ✅ 软件SPI统一方案
Adafruit_ST7789 tft(CS, DC, MOSI, SCLK, RST);  // 5参数 = 软件SPI

void fontSend(uint8_t dat) { /* bit-bang */ }
uint8_t fontRecv() { /* bit-bang */ }

void drawHanzi(...) {
    fontRead(...);           // 软件SPI读字库
    tft.drawRGBBitmap(...);  // 软件SPI写屏幕
}
```

**优点**：
- 无SPI冲突
- 任何GPIO都可以使用

**缺点**：
- 刷新速度慢（~1MHz等效）
- `fillScreen`约300ms

### 8.3 阶段3：硬件SPI统一期（推荐）

```cpp
// ✅ 硬件SPI统一方案
Adafruit_ST7789 tft(CS, DC, RST);  // 3参数 = 硬件SPI

void setup() {
    SPI.begin();  // 初始化硬件SPI，配置MISO
    tft.init(240, 240, SPI_MODE0);
}

bool fontRead(uint32_t addr) {
    SPI.beginTransaction(SPISettings(8MHz, MSBFIRST, SPI_MODE0));
    digitalWrite(FONT_CS, LOW);
    SPI.transfer(0x03);
    // ...
    digitalWrite(FONT_CS, HIGH);
    SPI.endTransaction();
}

void drawHanzi(...) {
    fontRead(...);           // 硬件SPI读字库
    tft.drawRGBBitmap(...);  // 硬件SPI写屏幕（40MHz）
}
```

**优点**：
- 刷新速度快（40MHz）
- `fillScreen`约23ms
- DMA支持

**缺点**：
- 依赖特定GPIO（ESP32-S3默认FSPI引脚）
- 需要理解`beginTransaction`机制

## 九、关键知识点总结

### 9.1 Adafruit_ST7789构造方式

| 参数数量 | SPI模式 | 适用场景 |
|---------|---------|----------|
| 3参数(CS, DC, RST) | 硬件SPI | 需要高速刷新 |
| 5参数(CS, DC, MOSI, SCLK, RST) | 软件SPI | 任意GPIO，或避免SPI冲突 |

### 9.2 SPI.begin()的破坏性

| 场景 | 能否调用SPI.begin() | 原因 |
|------|---------------------|------|
| 5参数构造(软件SPI) | **绝对不能** | 会把GPIO切换给硬件SPI，digitalWrite失效 |
| 3参数构造(硬件SPI) | **必须调用** | 需要配置MISO等引脚 |

### 9.3 ST7789 IPS屏特性

| 特性 | 说明 |
|------|------|
| 像素极性 | 与普通TFT相反 |
| 必须命令 | `INVON (0x21)`，即`invertDisplay(true)` |
| SCK敏感性 | CS高时SCK必须保持高，否则产生额外时钟脉冲 |
| 工作模式 | SPI Mode 3（CPOL=1, CPHA=1） |

### 9.4 GT30L32S4W字库

| 参数 | 值 |
|------|-----|
| 点阵尺寸 | 15×16（存储为16×16，每行2字节） |
| 基地址 | 0x2C9D0 |
| 每个字符 | 32字节 |
| Zone 1偏移 | `(MSB-0xA1)*94 + (LSB-0xA1)` |
| Zone 2偏移 | `(MSB-0xB0)*94 + (LSB-0xA1) + 846` |

### 10.1 分阶段验证

```cpp

    // 阶段1：验证屏幕（纯英文，不碰字库）

    // 阶段3：验证汉字显示
```

### 10.2 常用排查清单

| 现象 | 排查项 |
|------|--------|
| 屏幕完全空白 | BLK是否接高电平？`invertDisplay(true)`是否调用？ |
| 有背光但全黑/全白 | `invertDisplay(true)`是否遗漏？ |
| 汉字不显示 | 地址计算是否正确？`SPI.begin()`是否被错误调用？ |
| 字库返回全零 | CS2(GPIO9)和MISO(GPIO13)接线是否正确？ |
| 显示闪烁 | CS1和CS2是否同时被拉低？ |
| 颜色反相 | `invertDisplay(true)`是否调用？是否有SPI干扰？ |

### 10.3 使用串口调试


## 十一、最终推荐方案

### 11.1 方案选择

| 需求 | 推荐方案 | 代码 |
|------|----------|------|
| 最高性能 | 硬件SPI统一 | `st7789_hanzi_hwspi.ino` |
| 最简实现 | 软件SPI统一 | `st7789_hanzi_demo.ino` |
| 多SPI设备 | TFT_eSPI + 独立SPI | `chinese_font_display_tft_espi.ino` |

### 11.2 硬件SPI版（最终推荐）

```cpp
#include <Adafruit_GFX.h>
#include <Adafruit_ST7789.h>
#include <SPI.h>

Adafruit_ST7789 tft = Adafruit_ST7789(TFT_CS, TFT_DC, TFT_RST);
// 3参数 = 硬件SPI

void setup() {
    SPI.begin();  // 必须：配置MISO
    tft.init(240, 240, SPI_MODE0);
    tft.invertDisplay(true);  // IPS屏必须
}

bool fontRead(uint32_t addr) {
    SPI.beginTransaction(SPISettings(8MHz, MSBFIRST, SPI_MODE0));
    digitalWrite(FONT_CS, LOW);
    SPI.transfer(0x03);
    SPI.transfer(addr >> 16);
    SPI.transfer(addr >> 8);
    SPI.transfer(addr);
    for (int i = 0; i < 32; i++) fontBuf[i] = SPI.transfer(0x00);
    digitalWrite(FONT_CS, HIGH);
    SPI.endTransaction();
}

void drawHanzi(...) {
    fontRead(addr);
    // 解码到pixBuf...
    tft.drawRGBBitmap(x, y, pixBuf, 16, 16);  // 高效批量绘制
}
```

## 十二、经验教训

1. **理解硬件比写代码更重要**：ST7789对SCK边沿的敏感性、SPI Mode 3的时序要求，这些 datasheet 上的细节直接决定了代码能否工作。

2. **库的内部机制要了解**：Adafruit_ST7789的5参数和3参数构造看似只是参数数量不同，实则决定了完全不同的底层实现（软件SPI vs 硬件SPI）。

3. **分阶段验证**：先验证屏幕能亮，再验证字库能读，最后验证汉字能显示。不要试图一次性解决所有问题。

4. **编码问题要提前规避**：使用`\xHH`转义表示汉字，避免文件编码带来的不确定性。

5. **批量操作优于逐点操作**：`drawRGBBitmap`比`drawPixel`快20倍以上，这是嵌入式开发的基本优化原则。

6. **SPI总线冲突是隐形杀手**：`SPI.begin()`的破坏性不是立即显现的，它会让后续的`digitalWrite`静默失效，极难排查。
