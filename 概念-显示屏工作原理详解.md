# 显示屏工作原理详解：从 GPIO 电信号到像素发光

> 目标：理解 ST7789 显示屏**为什么这样接线**、**代码为什么这样写**、**屏幕为什么会亮**。  
> 适合：刚接触 SPI 显示屏、想知道"每一根线到底在干什么"的开发者。

---

## 一、核心问题：一块屏幕需要什么才能显示？

抛开所有技术细节，思考一个基本问题：**你怎样让远处的人知道你想画什么？**

你需要告诉他三件事：
1. **画什么**（颜色：红、绿、蓝）
2. **画在哪**（坐标：第几行第几列）
3. **什么时候画**（时序：先给坐标还是后给颜色）

屏幕和 ESP32 的关系也一样。**ESP32 是"画家"，ST7789 屏幕是"画布"**，它们之间需要一根线传颜色、一根线传坐标、一根线告诉对方"现在该画了"。

这就是所有接线的本质——**用物理电线传递"画什么、画在哪、什么时候画"三个信息**。

---

## 二、从物理到逻辑：10 根线各自在干什么

你的 ST7789 有 10 根线，可分为四组：

### 第 1 组：生存必需（电源）

| 线名 | 接到哪 | 为什么必须有 |
|------|--------|-------------|
| VCC | 3.3V | ST7789 芯片和液晶面板需要电才能工作 |
| GND | GND | 所有电信号的参考基准（0V），没有 GND 就不知道 3.3V 是什么意思 |

**思维要点**：任何电子设备，**VCC + GND 是生命线**。不接 GND，信号没有回路，芯片根本不知道你在说什么。

### 第 2 组：告诉屏幕"画什么"（SPI 数据总线）

| 线名 | GPIO | 全称 | 作用 |
|------|------|------|------|
| SDA | GPIO11 | MOSI | ESP32 → 屏幕：**发送数据**（颜色、坐标） |
| SCL | GPIO12 | SCK | ESP32 → 屏幕：**发送时钟**（告诉对方"现在读这一位"） |
| FSO | GPIO13 | MISO | 屏幕 → ESP32：**回读数据**（只有字库芯片用） |

**为什么是 SPI 协议？**

ST7789 屏幕需要**高速传输**（240×240 像素 = 57600 个点，每点 2 字节 = 115KB 一帧），I2C 太慢（最高 1MHz），SPI 可达 40MHz+。所以选择 SPI 是**性能驱动的**。

**SPI 工作的核心思维**：

```
ESP32 想说："我要在 (50, 80) 画一个红点"

它不会一次性说完整句话，而是拆成二进制：
- 命令字节：0x2C（"我要开始写颜色数据了"）
- 颜色数据：0xF800（"红色的 16bit 编码"）

每 1 bit 的传递都需要 SCK（时钟）配合：

SCK:  ▔\_/\_/\_/\_/\_/\_/\_/\_▔
       ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑
MOSI: 1   0   0   1   1   0   0   ...
      ↑
      每个时钟上升沿，屏幕读 MOSI 上的电平（高=1，低=0）
```

**关键认知**：SPI 没有真正的"语言"，只有**时序**——时钟线告诉对方"现在读"，数据线在那一刻的电平就是 0 或 1。

### 第 3 组：告诉屏幕"我在和谁说话"（片选）

| 线名 | GPIO | 作用 |
|------|------|------|
| CS1 | GPIO10 | **屏幕片选**：拉低 = "我在跟你说话" |
| CS2 | GPIO9 | **字库片选**：拉低 = "我在跟字库说话" |

**为什么需要片选？**

因为 SDA/SCL 是**总线**——两根线挂了两个设备（屏幕 + 字库）。如果 ESP32 在 SDA 上发数据，两个设备都会听到。**CS 线就是"点名"**——CS1 拉低时，只有屏幕会响应；CS2 拉低时，只有字库会响应。

```
总线结构：

         ESP32
           │
    SDA ───┼────────┬────────┐
           │        │        │
    SCL ───┼────────┼────────┤
           │        │        │
    CS1 ───┴────────┤        │    ← 只有屏幕听
           │        │        │
    CS2 ────────────┴────────┤    ← 只有字库听
           │        │        │
        ST7789   GT30L32S4W
        (屏幕)    (字库)
```

**思维要点**：任何时候**只能拉低一个 CS**，否则两个设备同时响应，数据会冲突。

### 第 4 组：控制信号

| 线名 | GPIO | 全称 | 作用 |
|------|------|------|------|
| DC | GPIO6 | Data/Command | **高=数据帧，低=命令帧**——告诉屏幕"我发的是颜色还是指令" |
| RES | GPIO7 | Reset | **低电平复位**——让屏幕芯片恢复到初始状态 |
| BLK | GPIO5 | Backlight | **高=开背光，低=关背光**——控制 LED 背光板的亮灭 |

**为什么需要 DC 线？**

ST7789 芯片接收到的每个字节都有两种含义：
- **命令**（0x01 = 软件复位、0x11 = 退出睡眠）
- **数据**（0xF800 = 红色像素值）

屏幕怎么区分？靠 DC 引脚的电平：

```
ESP32 想发"复位命令"：
  1. DC = LOW（告诉屏幕：这是命令）
  2. CS1 = LOW（选中屏幕）
  3. MOSI 发送 0x01（复位命令）
  4. CS1 = HIGH（释放屏幕）

ESP32 想发"红色像素"：
  1. DC = HIGH（告诉屏幕：这是数据）
  2. CS1 = LOW（选中屏幕）
  3. MOSI 发送 0xF800（红色）
  4. CS1 = HIGH（释放屏幕）
```

**为什么需要 RES 线？**

上电瞬间，ST7789 芯片内部的寄存器是随机的（就像电脑刚开机内存里是垃圾数据）。**RES 给一个低电平脉冲**，芯片内部电路会把所有寄存器恢复到出厂默认值，进入已知状态。

```
RES:  ▔▔▔▔▔\____/▔▔▔▔▔
             ↑
         低电平脉冲（>10μs）
         芯片复位，寄存器清零
```

**为什么需要 BLK 线？**

IPS 液晶面板本身不发光，它是**遮挡/透过背光**来显示颜色的。BLK 控制背光 LED 的亮灭——BLK=HIGH 时 LED 亮，你才能看到画面；BLK=LOW 时 LED 灭，屏幕全黑。

## 三、屏幕内部的"工作流程"

理解了接线，现在进入 ST7789 芯片内部，看它收到数据后发生了什么。

### ST7789 内部结构（简化）

```
┌─────────────────────────────────────────┐
│              ST7789 芯片                 │
│                                          │
│  ┌─────────┐    ┌─────────────┐        │
│  │ SPI 接口 │───▶│ 命令解析器   │        │
│  │ (SCK等)  │    │ (识别0x01等) │        │
│  └─────────┘    └──────┬──────┘        │
│                        │                │
│         ┌──────────────┼──────────┐    │
│         ▼              ▼          ▼    │
│    ┌────────┐    ┌─────────┐  ┌──────┐│
│    │ 睡眠控制│    │ 显示方向 │  │行列地址││
│    │ 寄存器 │    │ 寄存器   │  │寄存器 ││
│    └────────┘    └─────────┘  └──────┘│
│         │              │          │     │
│         └──────────────┼──────────┘     │
│                        ▼                │
│                 ┌────────────┐          │
│                 │ GRAM 显存  │          │
│                 │ 240×240×18bit│        │
│                 │ (每像素 18bit)│       │
│                 └──────┬─────┘          │
│                        │                │
│                 ┌──────▼──────┐        │
│                 │ 液晶驱动电路│        │
│                 │ (控制每个像│        │
│                 │  素的透光率)│       │
│                 └──────┬──────┘        │
│                        │                │
│                     液晶面板            │
│                     (物理像素)          │
└─────────────────────────────────────────┘
```

### 显示一帧画面的完整流程

**Step 1：初始化（Setup）**

上电后，ST7789 处于"睡眠模式"（省电），你需要用一系列命令唤醒它：

```cpp
tft.init(240, 240);  // 这一行内部做了什么？
```

Adafruit_ST7789 库的 `init()` 函数内部，实际发送了 20+ 条命令：

```
1. RES 低电平复位（>10ms）
2. 0x11 - 退出睡眠模式（Sleep Out）
3. 等待 120ms（芯片内部电路稳定需要时间）
4. 0x36 - 设置显示方向（MX/MY/MV 参数）
5. 0x3A - 设置像素格式（16bit 色深 = RGB565）
6. 0x21 - 打开显示反转（IPS 屏特性）
7. 0x13 - 打开普通显示模式
8. 0x29 - 打开显示（Display On）
```

**为什么需要这么多命令？**

ST7789 是一块通用驱动芯片，可以搭配不同尺寸的屏幕、不同的像素排列方向。**这些命令就是"配置"**——告诉芯片"你接的是 240×240 的屏、像素是 RGB565 格式、方向是正的"。

**Step 2：设置画在哪里（设置行列地址）**

ST7789 内部有一块显存（GRAM），大小是 240×240×18bit。你想画哪个像素，需要先告诉芯片"我要操作这块区域"：

```cpp
tft.fillRect(0, 0, 80, 80, ST77XX_RED);
// 内部实际发送：
// 0x2A - 设置列地址（X 范围：0~79）
// 0x2B - 设置行地址（Y 范围：0~79）
// 0x2C - "我要开始写颜色数据了"
```

**为什么先设地址再写数据？**

因为 GRAM 是**一维数组**，但屏幕是**二维平面**。芯片内部有一个"地址指针"，指向当前要写的 GRAM 位置。你设置好 X/Y 范围后，每发一个颜色字节，指针自动+1，直到填满你设定的区域。

```
GRAM 内存布局（240×240）：

行 0: [像素0] [像素1] [像素2] ... [像素239]
行 1: [像素0] [像素1] [像素2] ... [像素239]
...
行239:[像素0] [像素1] [像素2] ... [像素239]

地址指针从 (0,0) 开始，每写 2 字节（一个像素），指针+1
写完第 239 列，自动跳到下一行第 0 列
```

**Step 3：写颜色数据（像素值）**

ST7789 支持多种颜色格式，Arduino 库默认用 **RGB565**（16bit）：

```
RGB565 格式：
  红 5bit + 绿 6bit + 蓝 5bit = 16bit

红色：R=11111, G=000000, B=00000 → 0xF800
绿色：R=00000, G=111111, B=00000 → 0x07E0
蓝色：R=00000, G=000000, B=11111 → 0x001F
白色：R=11111, G=111111, B=11111 → 0xFFFF
黑色：R=00000, G=000000, B=00000 → 0x0000
```

**为什么是 RGB565 而不是 RGB888？**

因为 RGB565 只要 2 字节/像素，RGB888 要 3 字节/像素。对于 240×240 屏幕：
- RGB565：240×240×2 = 115,200 字节（一帧）
- RGB888：240×240×3 = 172,800 字节（一帧）

用 RGB565 可以**节省 33% 的传输时间和内存**。

**Step 4：芯片驱动液晶面板**

你写入 GRAM 的只是数字，真正让像素发光的是 ST7789 内部的**液晶驱动电路**。它会：

1. 读取 GRAM 中每个像素的 RGB 值
2. 通过 DAC 转换成模拟电压（0~3.3V）
3. 加到液晶分子两端，改变液晶排列角度
4. 液晶角度决定透光率，配合背光 LED 产生不同颜色

**关键认知**：你写的数字**不会直接发光**，它只是控制液晶分子"转多少角度"，最终颜色由"背光亮度 × 液晶透光率"决定。

## 四、代码思维的拆解：为什么这样写

### 代码 1：基础初始化

```cpp
Adafruit_ST7789 tft = Adafruit_ST7789(TFT_CS, TFT_DC, TFT_MOSI, TFT_SCLK, TFT_RST);
```

**为什么这么传参数？**

Adafruit_ST7789 类继承自 Adafruit_GFX，它内部需要知道：
- **CS**：哪个 GPIO 控制片选（和字库分时使用 SPI 总线）
- **DC**：哪个 GPIO 区分数据和命令
- **MOSI/SCLK**：SPI 数据线和时钟线
- **RST**：哪个 GPIO 控制复位

**思维要点**：这个构造函数**不实际发任何命令**，只是把 GPIO 编号存到类的成员变量里，供后续函数使用。

```cpp
void setup() {
  pinMode(TFT_BL, OUTPUT);
  digitalWrite(TFT_BL, HIGH);   // 背光开
```

**为什么先开背光？**

因为背光和 ST7789 芯片是**独立**的——芯片初始化需要几十毫秒，但背光 LED 点亮是瞬间的。先开背光，即使后续屏幕初始化花点时间，你也能看到变化（比如从全黑到显示内容）。

**如果先 `tft.init()` 再开背光？**

也可以，但你可能在初始化完成后才看到画面，无法判断初始化过程中屏幕是否有过反应（比如白屏一闪而过）。

```cpp
  tft.init(240, 240);
```

**内部发生了什么？**

1. `pinMode(RST, OUTPUT)` — 设置复位引脚为输出
2. `digitalWrite(RST, LOW)` — 拉低复位
3. `delay(10)` — 等待 10ms（芯片要求 >10μs）
4. `digitalWrite(RST, HIGH)` — 释放复位
5. 通过 SPI 发送 20+ 条初始化命令（见 Step 1）
6. 设置屏幕尺寸为 240×240

```cpp
  tft.setRotation(0);
```

**为什么需要设置旋转？**

ST7789 的 GRAM 是固定的（0,0 在左上角），但屏幕安装方向可能不同。`setRotation()` 通过设置 0x36 命令的 MX/MY/MV 参数，**改变行列地址的映射关系**：

```
rotation=0: (0,0) 在左上角，X 向右，Y 向下
rotation=1: (0,0) 在左下角，X 向上，Y 向右  ← 90度旋转
rotation=2: (0,0) 在右下角，X 向左，Y 向上  ← 180度旋转
rotation=3: (0,0) 在右上角，X 向下，Y 向左  ← 270度旋转
```

**关键认知**：旋转不是在"转画面"，而是**重新定义坐标系**。你的 `fillRect(0,0,80,80)` 在 rotation=0 和 rotation=1 下，实际写入的 GRAM 地址是不同的。

```cpp
  tft.fillScreen(ST77XX_BLACK);
```

**内部做了什么？**

1. 设置列地址范围：0 ~ 239（0x2A 命令）
2. 设置行地址范围：0 ~ 239（0x2B 命令）
3. 发送 0x2C（"我要写数据了"）
4. 循环发送 240×240 = 57600 个黑色像素值（0x0000）

**为什么用 `fillScreen()` 清屏而不是逐像素画？**

因为 `fillScreen()` 内部优化为**连续发送数据**，不需要每次设置地址。逐像素画需要每点都发 0x2A/0x2B，效率低 3 倍以上。

---

### 代码 2：画矩形

```cpp
  tft.fillRect(0, 0, 80, 80, ST77XX_RED);
```

**内部发生了什么？**

```
1. 设置列地址：Start=0, End=79    (0x2A)
2. 设置行地址：Start=0, End=79    (0x2B)
3. 发送 0x2C（"开始写数据"）
4. 循环 80×80 = 6400 次，每次发送 2 字节红色值（0xF800）
```

**为什么 `fillRect` 比画 6400 个像素快？**

因为 ST7789 支持**地址自增**。一旦设置了起始地址和结束地址，每发 2 字节颜色，内部地址指针自动+1，直到填满设定区域。**不需要每像素都重设地址**。

---

### 代码 3：显示文字

```cpp
  tft.setTextColor(ST77XX_WHITE);
  tft.setTextSize(2);
  tft.setCursor(10, 180);
  tft.println("ESP32-S3 OK!");
```

**内部发生了什么？**

Adafruit_GFX 库内置了一套 5×7 像素的小字体（英文字母和符号）。`println()` 会：

1. 读取字符串每个字符的 ASCII 码
2. 查内置字体表，得到该字符的 5×7 点阵数据
3. 对每个点阵中的"1"，在该位置画一个像素
4. 字符之间留 1 像素间距

**为什么只能显示英文？**

因为内置字体表只包含 ASCII 字符（0~127）。**中文字符（GB2312 编码）不在内置表中**，所以需要外挂字库芯片（GT30L32S4W）。

## 五、从"接线"到"像素"的完整思维链条

现在把全部内容串起来，形成一条**从物理连接到视觉呈现**的完整链条：

```
1. 【物理层】你接好 VCC/GND，芯片上电
   ↓
2. 【物理层】你拉高 BLK，LED 背光亮起（但液晶还是乱的，画面是白的或花的）
   ↓
3. 【协议层】代码执行 tft.init()
   ├── RES 低电平复位 → ST7789 寄存器清零
   ├── SPI 发送 0x11（退出睡眠）→ 芯片内部电路启动
   ├── SPI 发送 0x36/0x3A（配置方向/格式）→ 芯片知道"怎么解析后续数据"
   └── SPI 发送 0x29（打开显示）→ 芯片开始从 GRAM 刷新液晶面板
   ↓
4. 【协议层】代码执行 fillScreen(BLACK)
   ├── SPI 发送 0x2A/0x2B（设置地址范围 0~239, 0~239）
   ├── SPI 发送 0x2C（"我要写数据"）
   └── SPI 连续发送 57600 个 0x0000 → 填满 GRAM
   ↓
5. 【芯片层】ST7789 内部电路每 16.6ms（60Hz）读取一次 GRAM
   └── 把 GRAM 内容转换成电压，驱动液晶分子排列
   ↓
6. 【物理层】液晶分子根据电压改变透光角度
   └── 背光 LED 穿过液晶，被不同程度地遮挡
   ↓
7. 【视觉层】你看到一块黑色的屏幕
   ↓
8. 【协议层】代码执行 fillRect(0,0,80,80,RED)
   ├── SPI 发送 0x2A/0x2B（设置地址范围 0~79, 0~79）
   ├── SPI 发送 0x2C（"我要写数据"）
   └── SPI 连续发送 6400 个 0xF800 → 写入 GRAM 左上角区域
   ↓
9. 【芯片层】下一次刷新时，ST7789 读取到新的 GRAM 值
   └── 左上角区域的液晶分子改变角度，透过更多红光
   ↓
10.【视觉层】你看到左上角出现一个红色方块
```

## 六、关键思维总结

### 1. 接线思维的"三层模型"

| 层级 | 你在操作什么 | 你关心什么 |
|------|-------------|-----------|
| **物理层** | GPIO 高低电平、电线连接 | 电压对不对、有没有共地、线有没有接反 |
| **协议层** | SPI 时序、命令/数据格式 | CS 时机对不对、DC 状态对不对、时钟频率匹配吗 |
| **应用层** | `fillRect()`、`setCursor()` | 坐标对不对、颜色对不对、文本内容对不对 |

**debug 时从底层往上排查**：
- 屏幕不亮 → 先量 VCC/GND/BLK（物理层）
- 花屏 → 检查 SPI 时钟极性/相位（协议层）
- 位置不对 → 检查坐标值（应用层）

### 2. "先配置，后使用"原则

ST7789 不是"即插即用"的——上电后它处于随机状态，必须先发送一系列配置命令。这和所有外设一样：

```
使用任何外设的三步：
1. 初始化（配置工作模式）
2. 设置参数（地址、格式等）
3. 传输数据（真正干活）
```

### 3. 显存思维的"延迟刷新"

你写的颜色**不是立刻显示**的，而是写入 GRAM，等 ST7789 下一次刷新周期（60Hz = 每 16.6ms 一次）才会显示出来。

**关键认知**：`fillRect()` 执行完后，屏幕**不会立刻变化**。但因为 16.6ms 很短，人眼感觉不到延迟。

### 4. SPI 总线的"分时复用"

你的屏幕和字库共用 SDA/SCL，靠 CS1/CS2 切换。这就像多人共用一条电话线：

```
ESP32 想读字库：
  1. 确保 CS1=HIGH（屏幕不要听）
  2. CS2=LOW（字库听）
  3. 发 SPI 命令读字库
  4. CS2=HIGH（字库不要听）

ESP32 想写屏幕：
  1. 确保 CS2=HIGH（字库不要听）
  2. CS1=LOW（屏幕听）
  3. 发 SPI 命令写屏幕
  4. CS1=HIGH（屏幕不要听）
```

## 七、常见问题原理分析

### 问题 1：全白屏，背光亮

**根本原因**：ST7789 没有收到正确的初始化命令，处于上电默认状态。

**为什么默认是白的？**

ST7789 上电后默认进入**睡眠模式 + 显示关闭**。此时 GRAM 内容是随机的（大部分是 0xFFFF = 白色），且液晶驱动电路没有工作。但因为背光 LED 已经亮了，你会看到一片白。

**排查**：
1. 检查 CS1 接线（是否接反成 CS2）
2. 检查 DC 接线（是否接反）
3. 用逻辑分析仪抓 SPI 波形，看 0x11 命令是否发出

### 问题 2：花屏 / 颜色不对

**根本原因**：像素格式不匹配。

ST7789 支持 RGB565、RGB666、RGB444 等多种格式。如果代码用 RGB565 发送，但芯片被配置成 RGB666，每个像素的位解析就会错位，导致颜色错乱。

**解决**：`tft.init()` 内部已经配置好 RGB565，一般不需要手动改。如果花屏，检查初始化命令是否被正确发送（SPI 是否通）。

### 问题 3：方向不对

**根本原因**：`setRotation()` 改变了行列地址映射，但你的代码坐标假设了另一种方向。

**注意**：rotation 改变的是**坐标系原点方向**，不是旋转画面。你的 `(0,0)` 在 rotation=0 是左上角，在 rotation=1 是左下角。

## 八、进阶：字库芯片的工作原理

GT30L32S4W 是一颗 SPI Flash，里面烧录了 GB2312 编码的汉字点阵。读取一个汉字的流程：

```
1. 计算汉字在 GB2312 中的"区"和"位"
   例："中" = GB2312 编码 0xD6D0
       区 = 0xD6 - 0xA0 = 54
       位 = 0xD0 - 0xA0 = 48

2. 计算 Flash 地址
   addr = 0x2C9D0 + (区-1) * 94 + (位-1)) * 32
   （每个汉字 16×16 = 256bit = 32 字节）

3. 通过 SPI 发送读命令（0x03）+ 3 字节地址

4. 连续读取 32 字节点阵数据

5. 每 2 字节 = 1 行 16 像素，逐行绘制
```

**为什么需要单独的 CS2？**

因为字库芯片和 ST7789 屏幕是**两个独立的 SPI 设备**，共用 SDA/SCL 但各自有片选。读字库时 CS2 拉低，写屏幕时 CS1 拉低，两者互不干扰。
